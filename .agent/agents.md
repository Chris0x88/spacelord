# AI Agent Guide for Pacman CLI

Do not introduce regressions. Always read `data/pools.json` or `data/variants.json` for structural changes.

This repository is **Pacman**, a specialized Hedera swap engine built around SaucerSwap V2. It is strictly optimized for **Hedera Token Service (HTS)**, native HBAR interactions, and a premium CLI experience.

## ⚠️ CRITICAL: The Role of WHBAR
**WHBAR (Wrapped HBAR)** is NOT just another token. It is the **fundamental routing bridge** for the entire application.
- **Router Tool, Not Asset:** In this architecture, WHBAR is primarily used to interact with the SaucerSwap V2 pools (which are ERC20-based).
- **Auto-Wrapping:** The system must automatically wrap HBAR -> WHBAR before swapping and unwrap WHBAR -> HBAR after swapping.
- **Liquidity Source:** Almost all liquidity pairs on SaucerSwap are against WHBAR. If you break the WHBAR handling, you break the entire DEX integration.

## 🛡️ Protected Core Components
The following files contain highly specific, battle-tested logic for HTS interactions. **DO NOT REFACTOR OR DELETE** without explicit user instruction.

- **`pacman_transfers.py`**: Handles the specific nuances of HBAR vs HTS token transfers. Future agents often try to "simplify" this into generic EVM logic—**this will fail**.
- **`pacman_translator.py`**: The "Rosetta Stone" for token aliases. It manages the specific fuzzy matching required for user-friendly CLI commands.
- **`pacman_variant_router.py`**: The routing brain. It knows about the HTS vs ERC20 bridged variants. Standard routing logic does not apply here.

## 🛠 Repository Structure

### Core Logic
- `pacman_cli.py`: The Main Entry Point. Handles user intent, command parsing, and display.
- `pacman_app.py`: The Controller. Coordinates the Router, Executor, and Price Manager.
- `pacman_executor.py`: High-level execution logic (swaps/transfers), balance checks, and security.
- `pacman_price_manager.py`: The Oracle. Manages live pricing data normalized from raw sources.

### Data Architecture (Source of Truth)
- `data/pools.json`: **Static Registry** of verified liquidity pools (Contract IDs, Fees).
- `data/variants.json`: **Static Registry** of token variants (e.g., WBTC[hts] vs WBTC[lz]).
- `data/tokens.json`: Official Token Metadata (Names, Symbols, Decimals).
- `data/pacman_data_raw.json`: **Variable Data** cache for live reserves and prices (generated by `scripts/refresh_data.py`).

## ⚠️ Non-Negotiable Rules

### 1. Token Approvals (HTS vs EVM)
Standard EVM `approve()` calls often **fail** for HTS tokens due to Hedera's precheck requirements.
- **Must use HTS Logic:** Accessing specific HTS tokens usually requires associating the token with the account first.
- **Allowance Type:** We use `approve_hts_token` logic (or equivalent SDK wrappers) rather than raw generic ERC20 calls whenever possible for native assets.

### 2. Static Data First
Do **not** hardcode token addresses or pool IDs in Python files.
- Add structural changes to `data/pools.json` or `data/variants.json`.
- Add metadata changes to `data/tokens.json` or `data/aliases.json`.

### 3. Milliseconds for Deadlines
Hedera's V2 SwapRouter requires deadlines in **milliseconds**. Using seconds will result in immediate reverts.

### 4. HBAR Value Scaling
Transactions sending native HBAR must scale the `value` field to **18 pseudo-decimals (pseudo-Wei)** for the Hashio Relay.
`scaled_value = tinybars * 10**10`

### 5. Multicall Recipient Context
When swapping **Token → HBAR**, the `exactInput` step MUST specify the **Router Address** as the recipient. This allows the subsequent `unwrapWHBAR` call to access the funds within the Router's own context.
